tinymce 定制笔记

最近因为各方面的需求重新折腾了一段时间编辑器.最后还是在最熟悉的 tinymce 上做了定制.

demo 见: ekoneko.github.io/tinymce

style

之前和别人讨论编辑器,tinymce 被吐槽最多的就是…太丑了. 其实编辑器的好看与否主要还是图标之类的样式. 虽然有点自夸之嫌,不过个人觉得稍微调整了一点点样式, demo 中的样式就完爆官方默认样式了. 我所做的事情只有两件:

把 icon 缩小了一些

去掉渐变把背景变成了纯色

此外做了一个非插件的功能, demo 的例子如果出现滚动条, toolbar 会自动固定在浏览器顶部. 类似于 google drive 的效果. 也很简单啦, js 监听下 scroll 而已.

problem

插件方面的开发都已经轻车熟路. 这次遇到的最大问题是 tinymce 加载上问题.

tinymce 自己实现了一套自动加载的机制. 对于普通使用是正常的, 但当动态加载时,许多判断会出问题.

比如说找不到 baseurl, 不过可以手动指定

tinymce.baseURL = '/libraries/tinymce';

tinymce 会根据引入的 js 的 script 标签是 tinymce.js 还是 tinymce.min.js 去判断加载 plugin.js 还是 plugin.min.js . 动态加载时这里的判断会失效默认加载了 plugin.js, 这在真实项目中是不期望的. 解决方案也类似

tinymce.suffix = '.min';

===

另外补充一个 tinymce 的基本配置项的查询

https://gist.github.com/ekoneko/fdba6f69c86308d9421b

why choose

相对的 tinymce 挺臃肿的, 个人更加偏好轻量一些的编辑器. 之所以使用它主要的原因还是因为目的上是需要一个编辑器去完成某些东西, 而不是去寻找一个最好用的编辑器.

在比较稳定的编辑器中个人最熟的就是 tinymce 了吧, WordPress 在用,以及之前公司的项目也在用. 个人也有在 tinymce 上做过蛮多插件开发以及针对3.x 进行 debug 之类的事情.

要说 tinymce 的优点, 有官方团队维护保持基本的稳定性. 定制性很强, 我的所有改动完全没有去修改 tinymce 本身的任何代码.完全是在加载器中挂载的. 包括 skin 啦, theme 以及插件的调整.

angular 实践

我尝试着在公司的一个新项目上使用angularjs.

这是一个内部使用的管理平台. 使用的是一款基于 jqueryui 和 bootstrap 的 adminUI. 理论上这样的项目不太适合 angular:

jquery 和 angular 理念是违背的, jquery是在dom上的封装, 而angular提倡不在js中操作dom.

angular 和 jquery 一部分功能代码是重叠的, 一起加载稍微有点浪费.

但这些缺点对于实际的影响并不是很严重, 比起 MVVM 的特性, 选择 angular 的目的更主要是为了更好的结构化代码, 在teamer写出完全让人抓狂的js前先抑制这种可能性.

factory 数据封装

按照公司的传统, 大部分的展示界面都是通过列表来呈现的. 而几乎所有列表数据都有共同之处: 请求 page, size, 搜索条件, 返回 {state: true, data: [{…}, {…}], total: xxx} 格式的 json 对象. 过去只封装了后端的数据接收, 每次构建前端都要自己初始化一个表格,然后一个个重复事件. 代码重复率很高.

现在用 angular 的服务继承搞定这个问题了. 创建了一个 List 的抽象服务, 所有列表都从 list 上继承了基本的行为. 绝大部分场合下,只需要指定 baseUrl 和 filter(过滤数据), 再配合MVVM即可很轻松的渲染出一个列表.

ngRoute 路由

用 angular 的路由遇到了一点小问题. angular 介绍是构建单页面webapp的, 官方给出的路由方法需要一次性加载全部controller. 这一点有点接受不了, 而我则更希望实现按需加载. 用google到的方案解决了(http://ify.io/lazy-loading-in-angularjs/).

与 jquery 共存

既然必须用到jquery, 我也没少用. 利用 jquery 的 Ajax 替代 angular 原生的 http 服务省去了修改php后端处理请求的麻烦(angular原生的请求没办法被php直接从$_GET, $_POST中获取). 而且不晓得 angular 的 http 如何实现全局的处理.(应该用server的继承来搞定吧). 一些restful的事情, 比如当服务器返回301的状态码时让页面跳转是用jquery 的 ajaxsetup 去实现的. jqueryui 的一些组件(比如dialog)会在 body 最外层创建 dom,切换view时会带来不必要的麻烦. 统一在路由切换时干掉了.

小结

总体来看 angular 比当初听到的要简单许多. 学习曲线也很平缓. 虽然用起来遇到了一些各种各样的问题不过社区环境相对比较完善了, 可以很容易的发现问题并解决. 对于不需要兼容IE6, 且js交互比较复杂的项目建议使用.

angular 简介

angular 是 Google 开发的一款 MVVM 前端框架.

说到MVVM, M是 model, 可理解为数据, V是 view, 可理解为UI层. M to V & V to M, 也就是实现了一种数据与视图之间的关联模式.

看起来都只是一两行的代码, 但仔细想想就能感受到MVVM的涵义了:

让model与view完全分离开, js专注去实现数据的变化. 交互与显示则完全交由HTML或其他模板语言负责.

Angular 是我所知道的MVVM框架中体积最大的一个(源码20000+代码, 压缩后的体积也有100+kb). 当然也是功能最齐全的一个. 除了上述最核心的数据绑定外,还实现了模板, 路由, 控制器, 服务等功能. 主要是为单页面应用提供便利,当然传统web也无不可.

默认的浏览器兼容是到 IE9, 但官方给出了IE7,8的兼容性解决方案. 实现起来多加几个字代码即可, 除了不能自定义标签外没啥影响.

此外 Angular 的配套组件非常多. 值得一提的是Angular-Bootstrap 的 UI组件. 实现了grid啊, dialog啊,tips等等常用的UI组件, 使用起来可以少些很多代码.

为什么要用 Angular

说了这么多来总结一下 Angular.js 的优点

  1. 提供了一套优雅的代码方案, 团队开发中可作为一套编码规范

  2. 大幅度提高开发效率.

  3. 代码可读性高.

当然,这些都是相对的. 用Angular写出来的东西是不是高效优雅归根结底还是要看人… Angular作为工具只是提供了一种更加便利的方式

缺点

  1. 团队开发成本. 优点再多也要足够说服团队去用才行. 并不是人人都会angular, 学习一个新东西还是有成本的.

  2. 需要加载一个100kb的类库.

  3. Angular 提倡模型-视图分离, 但和其他类库并用时并不一定可以做到这点. 例如 jqueryUI.

  4. Angular 写的代码不能用通常的混淆方式进行压缩. 主要是 Angular的服务是依赖注入的方式加载的. 例如之前例子的
