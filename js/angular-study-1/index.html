angular 实践

我尝试着在公司的一个新项目上使用angularjs.

这是一个内部使用的管理平台. 使用的是一款基于 jqueryui 和 bootstrap 的 adminUI. 理论上这样的项目不太适合 angular:

jquery 和 angular 理念是违背的, jquery是在dom上的封装, 而angular提倡不在js中操作dom.

angular 和 jquery 一部分功能代码是重叠的, 一起加载稍微有点浪费.

但这些缺点对于实际的影响并不是很严重, 比起 MVVM 的特性, 选择 angular 的目的更主要是为了更好的结构化代码, 在teamer写出完全让人抓狂的js前先抑制这种可能性.

factory 数据封装

按照公司的传统, 大部分的展示界面都是通过列表来呈现的. 而几乎所有列表数据都有共同之处: 请求 page, size, 搜索条件, 返回 {state: true, data: [{…}, {…}], total: xxx} 格式的 json 对象. 过去只封装了后端的数据接收, 每次构建前端都要自己初始化一个表格,然后一个个重复事件. 代码重复率很高.

现在用 angular 的服务继承搞定这个问题了. 创建了一个 List 的抽象服务, 所有列表都从 list 上继承了基本的行为. 绝大部分场合下,只需要指定 baseUrl 和 filter(过滤数据), 再配合MVVM即可很轻松的渲染出一个列表.

ngRoute 路由

用 angular 的路由遇到了一点小问题. angular 介绍是构建单页面webapp的, 官方给出的路由方法需要一次性加载全部controller. 这一点有点接受不了, 而我则更希望实现按需加载. 用google到的方案解决了(http://ify.io/lazy-loading-in-angularjs/).

与 jquery 共存

既然必须用到jquery, 我也没少用. 利用 jquery 的 Ajax 替代 angular 原生的 http 服务省去了修改php后端处理请求的麻烦(angular原生的请求没办法被php直接从$_GET, $_POST中获取). 而且不晓得 angular 的 http 如何实现全局的处理.(应该用server的继承来搞定吧). 一些restful的事情, 比如当服务器返回301的状态码时让页面跳转是用jquery 的 ajaxsetup 去实现的. jqueryui 的一些组件(比如dialog)会在 body 最外层创建 dom,切换view时会带来不必要的麻烦. 统一在路由切换时干掉了.

小结

总体来看 angular 比当初听到的要简单许多. 学习曲线也很平缓. 虽然用起来遇到了一些各种各样的问题不过社区环境相对比较完善了, 可以很容易的发现问题并解决. 对于不需要兼容IE6, 且js交互比较复杂的项目建议使用.
